# 기능별 입력/출력 정리

이 문서는 기능을 확인하는 개발자가 "어떤 파일을 넣으면 어떤 결과물이 나오는지"를 바로 볼 수 있도록 정리한 문서입니다. 화면 기준으로 입력 파일, 처리 방식, 출력 위치와 파일명 규칙을 함께 적었습니다.

## 공통 실행

```powershell
cd C:\eGovFramework\workspace\atoz_ai_integrated
.\venv\Scripts\activate
python web_app.py --host 127.0.0.1 --port 8000
```

접속 주소:

```text
http://127.0.0.1:8000
```

공통 출력 위치:

| 구분 | 위치 |
| --- | --- |
| 임시 업로드/작업 파일 | `web_runtime\temp` |
| 결과 폴더 | `web_runtime\results` |
| 산출물 덤프 결과 | `web_runtime\results\folder-dumps` |
| 템플릿 반영 결과 | `web_runtime\results\template-builds` |
| 직접 업로드 폴더의 메타데이터 반영 결과 | `web_runtime\results\metadata-dumps` |
| 서버 로그 | `web_runtime\web_app.log` |

## 핵심 처리 규칙

### 표지 있음/없음 기준

| 파일 종류 | 표지 있음으로 보는 조건 | 표지 없음일 때 처리 |
| --- | --- | --- |
| Excel 계열 | 시트명에 `표지`가 포함된 시트가 있음 | 새 표지를 만들지 않고 입력 파일 복사본을 결과로 쓴다. |
| HWP/HWPX, PPT 계열 | 표지/문서 텍스트에서 프로젝트명, 문서명, 미리보기 텍스트 중 하나라도 읽힘 | 새 표지를 만들지 않고 입력 파일 복사본을 결과로 쓴다. |
| 표지 읽기 오류 | 파일을 읽다가 오류가 발생함 | `fallback`으로 처리하고 입력 파일 복사본을 결과로 쓴다. |

### 템플릿 기능의 결과 생성 규칙

템플릿 반영은 아래 조건을 모두 만족할 때만 서버 본문 템플릿과 서버 표지 템플릿을 조합한다.

- 입력 파일에 표지가 있다.
- 입력 파일이 문서관리표준의 산출물명과 매칭된다.
- `templates\개발산출물` 또는 `templates\관리산출물` 안에서 본문 템플릿을 찾는다.
- `templates\표지` 안에서 같은 문서 계열의 표지 템플릿을 찾는다.

| 조건 | 결과 소스 | 표지 처리 | 출력 파일명 |
| --- | --- | --- | --- |
| 표지 있음 + 표준 매칭 + 본문 템플릿 있음 + 표지 템플릿 있음 | 서버 본문 템플릿 | 서버 표지 템플릿을 삽입/교체하고 토큰 치환 | 표지 템플릿 파일명 토큰 기준 |
| 표지 있음 + 표준 매칭 + 본문 템플릿 있음 + 표지 템플릿 없음 | 입력 파일 복사본 | 기존 표지는 유지하고 프로젝트명/개정정보만 갱신 | 입력 파일 상대 경로 기준 |
| 표지 없음 + 표준 매칭 | 입력 파일 복사본 | 표지를 새로 만들지 않음 | 입력 파일 상대 경로 기준 |
| 표지는 있으나 표준 매칭 실패 | 입력 파일 복사본 | 가능한 경우 기존 표지의 프로젝트명/개정정보만 갱신 | 입력 파일 상대 경로 기준 |
| 표준 매칭은 됐으나 본문 템플릿 없음 | 입력 파일 복사본 | 가능한 경우 기존 표지의 프로젝트명/개정정보만 갱신 | 입력 파일 상대 경로 기준 |
| HWP/HWPX 입력 | 위 규칙과 동일 | 처리 후 최종 확장자는 `.hwpx` | `.hwp`도 `.hwpx`로 출력 |

### 템플릿 매칭 우선순위

문서관리표준 산출물 매칭은 입력 문서번호를 기준으로 하지 않는다. 기존 문서번호가 오래된 값이어도 산출물명/문서명/경로가 맞으면 매칭된다.

| 대상 | 매칭 순서 |
| --- | --- |
| 입력 파일 ↔ 문서관리표준 | 1. 표지 문서명, 2. 파일명 산출물명, 3. 경로 산출물명 |
| 입력 파일 ↔ 서버 본문 템플릿 | 1. 같은 상대 경로, 2. 같은 파일명, 3. 표준 ID, 4. 표준 산출물명, 5. 정규화 파일명, 6. 표지 문서명, 7. 같은 파일명 계열 |

### 요구사항 ID별 생성 규칙

| 기능 | 요구사항 ID를 읽는 곳 | 생성 규칙 |
| --- | --- | --- |
| 개발산출물 덤프 반영 | 요구사항 파일명 안의 `SFR-...` | 반영된 파일을 템플릿으로 삼아 요구사항 ID별 파일을 만든다. |
| 관리산출물 덤프 반영 | 제안요청서 요구사항목록표 주변의 `SFR-...` | 기존 파일명 뒤쪽에 `SFR-...` tail이 있는 파일만 ID 치환/복제 대상으로 본다. |
| 개발 템플릿 반영 | 요구사항 파일명 안의 `SFR-...` | 요구사항 ID가 여러 개면 출력 파일도 ID별로 여러 개 만든다. |
| 관리 템플릿 반영 | 제안요청서 본문에서 추출한 `SFR-...` | 요구사항 ID가 여러 개면 출력 파일도 ID별로 여러 개 만든다. |
| 요구사항 ID 없음 | 없음 | 단일 파일로 만들거나, 해당 요구사항별 자동 생성은 건너뛴다. |

제안요청서 텍스트 추출 중 표 셀/줄바꿈 때문에 `SFR OOO 13`처럼 공백으로 떨어진 조각은 하나의 ID로 조립하지 않는다. 실제 원문에 `SFR-OOO-13`처럼 하이픈이 있는 ID가 있으면 유효한 요구사항 ID로 유지한다. 다만 같은 문서 안에서 `SFR-001`, `SFR-002`, `SFR-013`처럼 다수 ID가 같은 형태인데 `SFR-OOO-13`처럼 다른 형태가 같은 숫자 또는 접두 관계로 함께 잡히면 추출 충돌 후보로 보고 제외한다. 관리/개발 템플릿 결과 리포트의 `요구사항 ID 추출 근거` 섹션에서 ID별 원본 파일, 매칭 텍스트, 주변 문맥, 제외 사유를 확인할 수 있다.

### 산출물 덤프 반영 규칙

| 상황 | 처리 |
| --- | --- |
| 원본 폴더 | 직접 수정하지 않는다. 항상 `folder-dumps\원본폴더명_v0.N` 복사본만 수정한다. |
| 매칭된 파일 | 문서번호, 문서명, 프로젝트명, 버전, 파일명을 표준 기준으로 갱신한다. |
| 표지 구조가 예상과 다름 | 표지 갱신은 `failed` 또는 `partial`로 기록하되, 가능한 경우 파일명 변경은 계속 진행한다. |
| 표준 산출물은 아니지만 프로젝트명만 다른 문서 | 파일명은 유지하고 프로젝트명/개정정보만 갱신한다. |
| 관리산출물 화면 | 최상위 파일은 반영 대상에서 제외하고 하위 폴더 문서 중심으로 처리한다. |
| 제외 폴더/노이즈 파일 | `bak`, `backup`, `백업`, `임시`, `temp`, `tmp`, `Thumbs.db`, `desktop.ini`, Office 임시 파일은 기본 제외한다. |

## 1. 개발산출물 매핑

화면/API:

- 화면: `/check.html`
- 미리보기 API: `POST /api/check`
- 반영 API: `POST /api/folder-apply`

입력:

| 입력 | 필수 | 허용 형식 | 설명 |
| --- | --- | --- | --- |
| 문서관리표준 | 필수 | `.pdf` | 산출물 ID, 산출물명, 폴더 경로 기준을 읽는다. |
| 개발산출물 폴더 | 필수 | 폴더 업로드 | 실제 검사할 개발산출물 파일 묶음이다. |
| 요구사항 파일명 | 선택 | 확장자 제한 없음 | 파일명에 포함된 `SFR-...` ID를 요구사항별 자동 생성에 쓴다. |
| 개정일자/작성자/승인자 | 선택 | 텍스트 | 반영 시 표지/개정이력 초깃값으로 쓴다. |

처리와 출력:

| 버튼 | 처리 | 출력 |
| --- | --- | --- |
| 확인하기 | 문서관리표준의 개발산출물 목록과 업로드 폴더 파일을 비교한다. | 화면에 매칭/미매칭/후보 점수 표시. 결과 파일은 만들지 않는다. |
| 덤프 후 반영 | 업로드 폴더를 `folder-dumps` 아래로 복사한 뒤 복사본만 수정한다. | `web_runtime\results\folder-dumps\원본폴더명_v0.N` 결과 폴더 생성 |

반영 결과 폴더 안에 생기는 것:

- 표준 산출물 ID/산출물명 기준으로 이름이 바뀐 문서
- 문서 표지 또는 머리말의 문서번호, 문서명, 프로젝트명, 버전 값
- 기본 버전 `v0.1`이 반영된 파일명과 문서 메타데이터
- `README_검수결과.md`
- 요구사항 파일을 넣은 경우 `README_요구사항별_자동생성.md`
- 요구사항 파일을 넣은 경우 `SFR-...` ID별로 복제된 산출물 파일
- 프로그램소스 대상이 있으면 요구사항 ID별 빈 폴더

파일명 예시:

| 넣은 파일 | 문서관리표준 기준 | 나오는 파일 예 |
| --- | --- | --- |
| `구_인터뷰결과서_v0.0.hwpx` | `MFDS-ADT-R0101-01`, `인터뷰결과서` | `MFDS-ADT-R0101-01-인터뷰결과서_v0.1.hwpx` |
| `사용자인터페이스설계서_SFR-OLD-001.hwpx` | `MFDS-ADT-D0301-01`, `사용자인터페이스설계서` + 요구사항 `SFR-ESS-001` | `MFDS-ADT-D0301-01-사용자인터페이스설계서_v0.1_SFR-ESS-001.hwpx` |

주의:

- 현재 코드 기준 `/api/folder-apply`는 ZIP 다운로드 토큰을 붙이지 않고 `dump_root` 결과 폴더 경로를 반환한다.
- `bak`, `backup`, `백업`, `임시`, `temp`, `tmp` 폴더와 `Thumbs.db`, `desktop.ini`, Office 임시 파일은 기본 처리 대상에서 제외된다.

## 2. 관리산출물 매핑

화면/API:

- 화면: `/management.html`
- 미리보기 API: `POST /api/check`
- 반영 API: `POST /api/folder-apply`

입력:

| 입력 | 필수 | 허용 형식 | 설명 |
| --- | --- | --- | --- |
| 문서관리표준 | 필수 | `.pdf` | 관리문서 ID 구간과 관리산출물명을 읽는다. |
| 관리산출물 폴더 | 필수 | 폴더 업로드 | 실제 검사할 관리산출물 파일 묶음이다. |
| 제안요청서 | 선택 | `.pdf`, `.hwp`, `.hwpx`, `.docx`, `.docm`, `.xlsx`, `.xlsm`, `.xltx`, `.xltm` | 요구사항목록표 주변의 `SFR-...` ID를 추출한다. |
| 개정일자/작성자/승인자 | 선택 | 텍스트 | 반영 시 표지/개정이력 초깃값으로 쓴다. |

출력:

- 개발산출물 매핑과 같은 구조의 `folder-dumps\원본폴더명_v0.N` 복사본
- 표준 관리문서 ID/관리산출물명 기준으로 이름과 표지가 갱신된 문서
- `README_검수결과.md`
- 제안요청서를 넣은 경우 요구사항 ID별 복제 대상 파일 생성 결과
- 제안요청서에서 추출한 ID는 기존 파일명 뒤쪽에 `SFR-...` tail이 있는 파일의 치환/복제에 사용된다.

예시:

| 넣은 파일 | 제안요청서에서 읽은 ID | 나오는 파일 예 |
| --- | --- | --- |
| `회의록_SFR-TMP-001.hwpx` | `SFR-IIL-003` | `표준ID-회의록_v0.1_SFR-IIL-003.hwpx` |
| `사업관리대장.xlsx` | `SFR-IIL-003` | 요구사항 ID별 복제 대상이 아니면 원본 성격 유지 |

## 3. 개발/관리 템플릿 매칭

화면/API:

- 개발: `/development-template.html`
- 관리: `/management-template.html`
- API: `POST /api/template-build`

입력:

| 입력 | 개발 화면 | 관리 화면 | 설명 |
| --- | --- | --- | --- |
| 문서관리표준 | 필수, `.pdf/.hwp/.hwpx` | 필수, `.pdf/.hwp/.hwpx` | 표준 ID, 표준 산출물명, 프로젝트명을 읽는다. |
| 산출물 입력폴더 | 필수 | 필수 | 표지를 읽을 기존 산출물 또는 본문 파일 묶음이다. |
| 요구사항 파일명 | 선택 | 없음 | 파일명에서 `SFR-...` ID를 읽어 개발산출물 요구사항별 출력에 쓴다. |
| 제안요청서 | 없음 | 선택 | 본문에서 `SFR-...` ID를 읽어 관리산출물 요구사항별 출력에 쓴다. |
| 개정일자/작성자/승인자 | 선택 | 선택 | 표지 토큰과 개정이력에 반영된다. |
| 서버 템플릿 폴더 | 자동 사용 | 자동 사용 | 개발은 `templates\개발산출물`, 관리는 `templates\관리산출물`을 사용한다. |
| 표지 템플릿 | 자동 사용 | 자동 사용 | `templates\표지\{{DOCUMENT_ID}}-{{DOCUMENT_TITLE}}_{{VERSION}}_{{REQUIREMENT_IDS}}.*`를 사용한다. |

버튼별 출력:

| 버튼 | 처리 | 출력 |
| --- | --- | --- |
| 템플릿 매칭 | 표준 ID, 입력 산출물 표지, 서버 본문 템플릿을 매칭한다. | 화면 표만 생성. 결과 파일 없음 |
| 덤프 후 반영 | 매칭 결과를 실제 파일로 만든다. | `web_runtime\results\template-builds\development-...` 또는 `management-...` |

반영 결과 폴더 안에 생기는 것:

- 매칭된 본문 템플릿과 표지 템플릿을 조합한 출력 파일
- 입력 파일에 표지가 있으면 프로젝트명/개정일자/작성자/승인자 갱신
- HWP/HWPX 계열은 최종 `.hwpx`로 출력
- 요구사항 ID가 여러 개면 ID별 파일이 각각 생성됨
- `TEMPLATE_BUILD_REPORT.md`
- 다운로드 응답에는 개별 파일 링크와 `{결과폴더명}_template-output.zip` 링크가 붙음

파일명 예시:

| 조건 | 나오는 파일 예 |
| --- | --- |
| 개발 템플릿 + 요구사항 없음 | `MFDS-ADT-R0101-01-인터뷰계획서_v0.1_.hwpx` 또는 템플릿 상대 경로 기준 파일 |
| 개발 템플릿 + `SFR-001`, `SFR-002` | `MFDS-ADT-D0301-01-사용자인터페이스설계서_v0.1_SFR-001.hwpx`, `..._SFR-002.hwpx` |
| 엑셀 본문 템플릿 | 표지 시트가 삽입 또는 교체된 `.xlsx/.xlsm` |
| PPT 본문 템플릿 | 표지 슬라이드가 삽입 또는 교체된 `.pptx/.pptm` |

## 4. 메타데이터 반영

화면/API:

- 화면: `/metadata.html`
- 미리보기 API: `POST /api/metadata-preview`
- 반영 API: `POST /api/metadata-apply`

입력 방식은 두 가지다.

| 방식 | 넣는 것 | 설명 |
| --- | --- | --- |
| 기존 결과 폴더 후처리 | `dump_root` 경로 + WBS + 문서관리표준 | `/check.html` 또는 `/management.html` 반영 결과 폴더를 그대로 후처리한다. |
| 폴더 직접 업로드 | 산출물 폴더 + WBS + 문서관리표준 | 업로드 폴더를 새 메타데이터 덤프로 복사한 뒤 처리한다. |

입력 파일:

| 입력 | 필수 | 허용 형식 | 설명 |
| --- | --- | --- | --- |
| WBS | 필수 | `.xlsx`, `.xlsm` | 산출물별 담당자, 개정일자 등을 읽는다. |
| 문서관리표준 | 필수 | `.pdf`, `.hwp`, `.hwpx` | 승인자/작성자 기준값과 산출물 정보를 읽는다. |
| 결과 폴더 경로 또는 산출물 폴더 | 필수 | 폴더 | 실제 메타데이터를 반영할 문서 묶음이다. |
| 제외 경로 | 선택 | 텍스트 | 반영에서 제외할 상대 경로 목록이다. |

출력:

| 버튼 | 출력 |
| --- | --- |
| 확인하기 | matched, ambiguous, unmatched 대상 목록. 파일 생성 없음 |
| 반영하기 + 기존 `dump_root` | 기존 결과 폴더 안 문서에 메타데이터 반영, `{dump_root.name}_메타데이터_결과.zip` 다운로드 |
| 반영하기 + 폴더 직접 업로드 | `web_runtime\results\metadata-dumps\원본폴더명_metadata_v0.N` 생성, ZIP 다운로드 |

반영되는 값:

- 표지 작성자/승인자/문서번호 관련 값
- 개정이력의 개정일자, 작성자, 담당자 계열 값
- WBS 기준 담당자와 문서관리표준 기준 승인자

## 8. CLI 산출물 매핑/반영

대상:

- `folder_apply.py`

입력과 출력:

| 명령 | 넣는 것 | 나오는 것 |
| --- | --- | --- |
| `--check-only` | 문서관리표준 PDF + 산출물 폴더 | 콘솔 매칭 요약. 파일 생성 없음 |
| `--check-only --json` | 문서관리표준 PDF + 산출물 폴더 | 전체 JSON 매칭 결과 |
| 반영 실행 | 문서관리표준 PDF + 산출물 폴더 + `--dump` 결과 부모 폴더 | `--dump` 아래 버전 덤프 폴더와 갱신 문서 |

예시:

```powershell
python folder_apply.py `
  --check-only `
  --standard "C:\path\문서관리표준.pdf" `
  --folder "C:\path\산출물폴더"
```

```powershell
python folder_apply.py `
  --standard "C:\path\문서관리표준.pdf" `
  --folder "C:\path\산출물폴더" `
  --dump "C:\path\결과폴더"
```

## 확인할 때 자주 헷갈리는 지점

- 매핑 화면의 문서관리표준은 현재 업로드 코드 기준 PDF만 허용한다.
- 템플릿 화면의 문서관리표준은 PDF/HWP/HWPX를 허용한다.
- `확인하기`, `템플릿 매칭`, `메타데이터 확인하기`는 미리보기라서 결과 파일을 만들지 않는다.
- `덤프 후 반영`, `반영하기`, QA 생성 버튼을 눌러야 실제 파일이 생성된다.
- 산출물 매핑 반영은 원본 폴더를 직접 수정하지 않고 `folder-dumps` 아래 복사본만 수정한다.
- QA 폴더 일괄 생성은 별도 다운로드 파일을 만드는 기능이 아니라 기존 `dump_root` 안의 QA 템플릿 파일을 교체 배치하는 기능이다.
