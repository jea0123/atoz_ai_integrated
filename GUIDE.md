# AI Integrated QA Tool 사용 가이드

이 문서는 이 프로젝트를 처음 보는 사람이 설치, 실행, 주요 기능 사용, 결과 확인까지 바로 따라갈 수 있도록 정리한 안내서입니다.

## 1. 프로젝트 한눈에 보기

이 도구는 산출물 폴더와 문서관리표준을 대조하고, QA 관련 산출물을 생성하거나 갱신하는 웹 기반 작업 도구입니다.

주요 화면은 3개입니다.

| 화면 | 주소 | 용도 |
| --- | --- | --- |
| 개발산출물 매핑 확인 | `/check.html` | 문서관리표준의 산출물 코드 범위와 개발산출물 폴더를 비교하고, 복사본에 문서번호/문서명 등을 반영합니다. |
| 관리산출물 매핑 확인 | `/management.html` | 문서관리표준의 관리문서 ID 범위와 관리산출물 폴더를 비교하고, 복사본에 문서번호/문서명 등을 반영합니다. |
| 산출물 메타데이터 설정 | `/metadata.html` | WBS 기준으로 작성자, 개정일자, 승인자 같은 문서 메타데이터를 확인하고 반영합니다. |
| 테스트 케이스/시나리오 생성 | `/qa.html` | 화면설계서와 기존 QA 양식을 기반으로 단위시험/통합시험 산출물을 생성하고 배치합니다. |

## 2. 실행 전 준비

### 필수 환경

- Windows
- Python 3.12 권장
- 한글 프로그램 설치
- Ollama API 접근 가능 환경
- 프로젝트 의존성 설치

HWP/HWPX 문서 처리에는 `pyhwpx`와 한글 COM 자동화가 사용됩니다. 따라서 한글 프로그램이 설치된 Windows 환경에서 실행해야 합니다.

### Ollama 설정

프로젝트 루트에 `.env` 파일을 만들고 다음 값을 설정합니다.

```env
OLLAMA_BASE_URL=http://127.0.0.1:11434
OLLAMA_MODEL=exaone3.5:7.8b
```

`OLLAMA_BASE_URL`에는 `/api/chat` 또는 `/api/generate`를 붙이지 않습니다. 서버 코드가 필요한 API 경로를 내부에서 조합합니다.

Ollama 설정이 없으면 산출물 매핑은 규칙 기반으로만 동작합니다. QA 생성 기능은 모델 호출이 필요하므로 실패할 수 있습니다.

## 3. 설치

PowerShell에서 프로젝트 루트로 이동합니다.

```powershell
cd C:\eGovFramework\workspace\atoz_ai_integrated
```

가상환경을 만들고 활성화합니다.

```powershell
python -m venv venv
.\venv\Scripts\activate
```

필요한 패키지를 설치합니다.

```powershell
pip install -r requirements.txt
```

## 4. 실행

가상환경이 활성화된 상태에서 서버를 시작합니다.

```powershell
python web_app.py --host 127.0.0.1 --port 8000
```

브라우저에서 다음 주소를 엽니다.

```text
http://127.0.0.1:8000
```

기본 접속 화면은 산출물 매핑 확인 화면입니다.

서버를 외부 PC에서 접속해야 한다면 host를 `0.0.0.0`으로 실행할 수 있습니다.

```powershell
python web_app.py --host 0.0.0.0 --port 8000
```

## 5. 산출물 매핑 확인 사용법

주소: `http://127.0.0.1:8000/check.html`

이 화면은 문서관리표준에 정의된 산출물 목록과 실제 산출물 폴더 안의 파일을 비교합니다. 확인만 할 수도 있고, 산출물 폴더 복사본을 만든 뒤 문서번호/문서명/프로젝트명/개정이력을 반영할 수도 있습니다.

### 입력 파일

| 입력 | 필수 | 설명 |
| --- | --- | --- |
| 문서관리표준 | 필수 | PDF 파일만 허용됩니다. 산출물 ID, 산출물명, 경로 템플릿 등을 읽습니다. |
| 검사 폴더 | 필수 | 대조할 산출물 폴더입니다. 브라우저에서 폴더 단위로 업로드합니다. |
| 요구사항 파일명 | 선택 | 파일명에 포함된 `SFR-...` 요구사항 ID를 기준으로 요구사항별 산출물 복제 생성을 할 때 사용합니다. |
| 개정연도/작성자/승인자 | 선택 | 덤프 후 반영 시 표지와 개정이력 영역에 반영됩니다. |

검사 폴더를 선택하지 않으면 로컬 개발용 기본 폴더인 `data\테스트`를 찾습니다. 이 폴더는 보통 Git에 포함되지 않으므로 일반 사용자는 검사 폴더를 직접 선택하는 것이 안전합니다.

### 확인하기

1. 문서관리표준 PDF를 선택합니다.
2. 검사할 산출물 폴더를 선택합니다.
3. 필요하면 `미매칭만 AI 확인` 옵션을 켜거나 끕니다.
4. `확인하기`를 누릅니다.
5. 결과 표에서 산출물별 매칭 후보, 대상 파일, 점수를 확인합니다.

매칭은 기본적으로 파일명, 경로, 문서 내부 제목 같은 규칙을 우선 사용합니다. 산출물 ID는 바뀔 수 있으므로 기존 파일 매칭 기준으로 쓰지 않습니다. Ollama가 설정되어 있고 AI 옵션이 켜져 있으면 규칙으로 찾지 못한 항목을 AI로 보강합니다.

### 덤프 후 반영

`덤프 후 반영`은 원본 폴더를 직접 수정하지 않습니다. 먼저 결과 폴더 아래에 복사본을 만들고, 그 복사본에만 변경을 적용합니다.

기본 결과 위치는 다음과 같습니다.

```text
web_runtime\results\folder-dumps
```

결과 폴더명은 원본 폴더명 뒤에 버전 번호가 붙습니다.

```text
원본폴더명_v0.1
원본폴더명_v0.2
```

반영 후 결과 폴더 안에는 검수 요약 파일도 생성됩니다.

```text
README_검수결과.md
```

요구사항 파일명을 함께 올린 경우에는 요구사항별 자동 생성 결과 요약도 생성될 수 있습니다.

```text
README_요구사항별_자동생성.md
```

산출물 매핑 결과 폴더는 이후 `/metadata.html`과 `/qa.html`에서 그대로 이어서 사용할 수 있습니다. 화면의 링크 또는 자동 입력된 결과 폴더 경로를 활용하면 됩니다.

## 6. 관리산출물 매핑 확인 사용법

주소: `http://127.0.0.1:8000/management.html`

이 화면은 개발산출물이 아니라 관리산출물 전용입니다. 문서관리표준에서 `관리문서 ID` 구간만 읽고, 산출물명, 파일명, 폴더 기준으로 관리산출물 폴더의 파일을 매칭합니다.

입력은 단순합니다.

| 입력 | 필수 | 설명 |
| --- | --- | --- |
| 문서관리표준 | 필수 | PDF 파일만 허용됩니다. `관리문서 ID` 구간을 읽습니다. |
| 관리산출물 폴더 | 필수 | 대조할 관리산출물 폴더입니다. 개발 화면의 `data\테스트` 기본 폴더 fallback을 사용하지 않습니다. |
| 제안요청서 | 선택 | 요구사항목록표 안의 `SFR-...` 고유번호를 읽어 반영된 관리산출물을 요구사항 ID별로 복제 생성합니다. |
| 개정연도/작성자/승인자 | 선택 | 덤프 후 반영 시 표지와 개정이력 영역에 반영됩니다. |

사용 순서는 개발산출물 화면과 같습니다.

1. 문서관리표준 PDF를 선택합니다.
2. 관리산출물 폴더를 선택합니다.
3. 요구사항 ID별 파일이 필요하면 제안요청서를 함께 선택합니다.
4. `확인하기`로 매칭 결과를 검토합니다.
5. 문제가 없으면 `덤프 후 반영`으로 복사본에 문서번호/문서명을 반영합니다.

제안요청서가 올라온 경우 관리산출물은 이번에 반영된 파일을 기준으로 요구사항 ID별 복제본을 만듭니다. QA 생성과는 연결하지 않습니다.

## 7. 산출물 메타데이터 설정 사용법

주소: `http://127.0.0.1:8000/metadata.html`

이 화면은 WBS와 문서관리표준을 기준으로 산출물 문서의 작성자, 개정일자, 승인자 정보를 확인하고 반영합니다.

### 입력 파일

| 입력 | 필수 | 설명 |
| --- | --- | --- |
| 산출물 매핑 결과 폴더 | 선택 | `/check.html`에서 만든 결과 폴더 경로입니다. 이 값을 쓰면 산출물 폴더를 다시 업로드하지 않아도 됩니다. |
| 산출물 폴더 | 조건부 필수 | 결과 폴더 경로를 쓰지 않을 때 직접 업로드합니다. |
| WBS 파일 | 필수 | `.xlsx`, `.xlsm` 파일을 지원합니다. |
| 문서관리표준 | 필수 | `.pdf`, `.hwp`, `.hwpx` 파일을 지원합니다. 승인자 기준값을 읽습니다. |

### 확인하기

1. 산출물 매핑 결과 폴더 경로를 확인하거나 산출물 폴더를 직접 선택합니다.
2. WBS 파일을 선택합니다.
3. 문서관리표준 파일을 선택합니다.
4. `확인하기`를 누릅니다.
5. 결과 표에서 반영 가능, 확인 필요, 미매칭 상태를 확인합니다.

### 덤프 후 반영

`덤프 후 반영`을 누르면 메타데이터가 반영된 결과를 만들고 ZIP 다운로드 링크를 제공합니다. 결과에는 매칭에서 제외한 항목은 반영하지 않습니다.

반영 기준은 화면 기준으로 다음과 같습니다.

| 항목 | 기준 |
| --- | --- |
| 작성자 | WBS 담당 |
| 개정일자 | WBS 시작일 |
| 승인자 | 문서관리표준 작성자 |
| 버전 | v0.1 기준 업데이트 |

## 8. 테스트 케이스/시나리오 생성 사용법

주소: `http://127.0.0.1:8000/qa.html`

이 화면은 산출물 매핑 결과 폴더를 기준으로 QA 산출물을 생성하고 기존 파일 위치에 배치합니다. 개별 파일만 생성하는 도구도 함께 제공합니다.

### 폴더 기준 일괄 생성 및 배치

권장 흐름은 `/check.html`에서 먼저 `덤프 후 반영`을 실행한 뒤, 생성된 산출물 매핑 결과 폴더를 기준으로 QA 생성을 실행하는 것입니다.

필요한 5종 문서 세트는 요구사항 ID 기준으로 묶입니다.

| 역할 | 지원 확장자 | 설명 |
| --- | --- | --- |
| 사용자인터페이스설계서 | `.hwp`, `.hwpx`, `.pdf` | 단위시험 케이스 생성의 입력입니다. |
| 단위시험케이스 | `.hwpx` | 생성 결과로 교체될 기존 양식입니다. |
| 단위시험결과서 | `.hwpx` | 단위시험 케이스 결과를 기반으로 교체됩니다. |
| 통합시험시나리오 | `.xlsx` | 통합시험 시나리오 생성의 기존 양식입니다. |
| 통합시험결과서 | `.xlsx` | 통합시험 결과서 생성의 기존 양식입니다. |

사용 순서는 다음과 같습니다.

1. `산출물 매핑 결과 폴더`에 `/check.html` 결과 폴더가 들어 있는지 확인합니다.
2. 다른 위치의 파일을 써야 하면 `산출물 폴더`를 직접 선택하거나 `문서별 위치 따로 지정`을 엽니다.
3. `생성 및 배치`를 누릅니다.
4. 결과 영역에서 요구사항별 처리 결과와 배치된 파일을 확인합니다.

자동 배치는 기존 파일을 기준 위치에서 교체합니다. 내부 로직은 `bak`, `backup`, `백업`, `원본` 폴더를 후보 탐색에서 제외합니다.

### 개별 생성 도구

`개별 생성 도구 열기`를 펼치면 파일 단위 생성도 할 수 있습니다.

#### 단위시험케이스 생성

입력:

- 기존 단위시험 케이스 `.hwpx`
- 사용자인터페이스설계서 `.hwp`, `.hwpx`, `.pdf`

결과:

- 단위시험 케이스 XLSX
- 단위시험 케이스 HWPX

#### 통합시험시나리오 생성

입력:

- 기존 통합시험 시나리오 `.xlsx`
- 단위시험 케이스 `.xlsx`
- 사용자인터페이스설계서 `.hwp`, `.hwpx`, `.pdf`

결과:

- 통합시험 시나리오 XLSX

개별 생성 결과 파일은 다운로드 후 임시 폴더에서 정리됩니다.

## 9. CLI 사용

웹 화면 없이 산출물 매핑 확인 또는 덤프 반영만 실행할 때는 `folder_apply.py`를 사용할 수 있습니다.

### 매핑 확인만 실행

```powershell
python folder_apply.py `
  --check-only `
  --standard "C:\path\문서관리표준.pdf" `
  --folder "C:\path\산출물폴더"
```

JSON 전체 결과가 필요하면 `--json`을 추가합니다.

```powershell
python folder_apply.py `
  --check-only `
  --json `
  --standard "C:\path\문서관리표준.pdf" `
  --folder "C:\path\산출물폴더"
```

### 덤프 후 반영까지 실행

```powershell
python folder_apply.py `
  --standard "C:\path\문서관리표준.pdf" `
  --folder "C:\path\산출물폴더" `
  --dump "C:\path\결과폴더"
```

자주 쓰는 옵션은 다음과 같습니다.

| 옵션 | 설명 |
| --- | --- |
| `--threshold` | 매칭 점수 기준값입니다. |
| `--match-mode rule_only` | 규칙 매칭만 사용합니다. |
| `--match-mode rule_ai_fallback` | 규칙 매칭 후 미매칭만 AI로 확인합니다. |
| `--match-mode ai_first` | AI 매칭을 우선 사용합니다. Ollama 설정이 필요합니다. |
| `--project-title` | 문서관리표준에서 읽은 프로젝트명을 수동으로 덮어씁니다. |
| `--ignore-folders` | 제외할 폴더명을 쉼표로 지정합니다. |
| `--transparent-folders` | 경로 비교에서 투명 폴더처럼 취급할 폴더명을 쉼표로 지정합니다. |
| `--map-only-under` | 특정 경로 prefix 아래만 스캔합니다. |

## 10. 생성 파일과 로그

실행 중 생성되는 파일은 대부분 `web_runtime` 아래에 만들어집니다.

```text
web_runtime\
  temp\                 임시 업로드 및 QA 생성 작업 폴더
  results\              다운로드 결과와 덤프 결과
  web_app.log           서버 로그
```

`web_runtime`은 Git에 포함하지 않는 런타임 폴더입니다. 서버 시작 시 일부 중단 요청의 임시 폴더는 자동 정리됩니다.

  web_uploads.py                multipart 업로드 파싱 및 안전한 파일 저장

## 12. 문제 해결

### 서버가 실행되지 않을 때

- 가상환경이 활성화되어 있는지 확인합니다.
- `pip install -r requirements.txt`가 완료되었는지 확인합니다.
- 같은 포트가 이미 사용 중이면 다른 포트로 실행합니다.

```powershell
python web_app.py --host 127.0.0.1 --port 8010
```

### 한글 문서 처리 중 오류가 날 때

- 한글 프로그램이 설치되어 있는지 확인합니다.
- 열려 있는 한글 프로세스가 문서를 잠그고 있으면 종료 후 다시 시도합니다.
- HWP/HWPX 파일이 손상되었거나 암호화되어 있지 않은지 확인합니다.

### QA 생성이 실패할 때

- `.env`의 `OLLAMA_BASE_URL`과 `OLLAMA_MODEL`을 확인합니다.
- Ollama 서버가 실행 중인지 확인합니다.
- 입력 파일에 화면 ID, 요구사항 ID, 처리 흐름이 충분히 포함되어 있는지 확인합니다.
- 폴더 기준 일괄 생성은 요구사항 ID 기준으로 5종 문서 세트를 찾지 못하면 중단될 수 있습니다.

### 폴더 업로드 결과가 비어 있을 때

- 브라우저에서 폴더 선택을 지원하는지 확인합니다.
- 실제 파일이 없는 빈 폴더는 브라우저 업로드에 포함되지 않을 수 있습니다.
- `Thumbs.db`, `desktop.ini`, Office 임시 파일 같은 노이즈 파일은 자동 제외됩니다.

### 매칭 결과가 기대와 다를 때

- 문서관리표준의 산출물명과 실제 파일명/문서 제목이 비슷한지 확인합니다.
- `미매칭만 AI 확인` 옵션을 켜고 다시 시도합니다.
- CLI 사용 시 `--threshold` 값을 조정해 봅니다.
- `bak`, `backup`, `백업`, `temp`, `tmp` 계열 폴더는 기본 정책상 제외될 수 있습니다.

## 13. 권장 작업 순서

처음 사용하는 경우 다음 순서로 진행하는 것이 가장 단순합니다.

1. `.env`를 설정하고 서버를 실행합니다.
2. 개발산출물은 `/check.html`, 관리산출물은 `/management.html`에서 문서관리표준과 대상 폴더를 업로드합니다.
3. `확인하기`로 매칭 결과를 검토합니다.
4. 문제가 없으면 `덤프 후 반영`으로 복사본 결과 폴더를 만듭니다.
5. `/metadata.html`에서 WBS와 문서관리표준을 넣고 메타데이터를 확인/반영합니다.
6. `/qa.html`에서 산출물 매핑 결과 폴더를 기준으로 QA 산출물을 생성 및 배치합니다.
7. 결과 폴더의 README와 서버 로그를 확인합니다.
