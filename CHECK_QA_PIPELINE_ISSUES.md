# check.html - qa.html 연계 이슈 메모

작성일: 2026-05-19

## 배경

현재 `check.html`의 `덤프 후 반영` 흐름은 검사 대상 폴더를 서버 런타임 영역으로 복사한 뒤, 복사본에 문서관리표준 기반 변경사항을 반영하고, 최종 결과를 ZIP 파일로 제공하는 구조다.

향후에는 이 결과물을 다시 `qa.html`에서 사용하여 QA 산출물을 생성하고, 이후 WBS 기반 작성자/최초 작성일/개정이력 정보를 추가하는 후속 작업이 필요하다.

## 이슈 사항

### 1. ZIP 생성을 제거할 경우 복사폴더가 계속 남을 수 있음

ZIP 파일을 만들지 않고 복사폴더를 그대로 유지하는 방식으로 변경하면, 기존처럼 복사폴더가 자동 삭제되지 않는다.

따라서 사용자가 `덤프 후 반영`을 여러 번 실행할 경우 실행 횟수만큼 복사폴더가 계속 생성될 수 있으며, 장기적으로는 런타임 결과 폴더에 불필요한 파일이 누적될 가능성이 있다.

## 사용자 판단

### 2. 실제 사용상 반복 실행 가능성은 낮음

다만 이 프로그램은 실제 업무 흐름상 동일 대상에 대해 여러 번 반복 실행하기보다는, 최종 산출물 정리 목적으로 한 번 실행될 가능성이 높다고 판단된다.

따라서 복사폴더가 계속 누적되는 문제는 구조적으로는 존재하지만, 실사용 빈도를 고려하면 치명적인 문제는 아닐 수 있다.

### 3. ZIP을 다시 풀어서 QA 작업을 시작하는 방식은 비효율적임

현재 구조처럼 결과물을 ZIP으로 제공한 뒤, 사용자가 다시 압축을 풀고, 그 압축 해제 폴더를 기준으로 `qa.html` 작업을 이어가는 흐름은 사용자 관점에서 번거롭고 비효율적이다.

특히 `check.html` 결과가 곧바로 `qa.html`의 입력이 되어야 하는 구조라면, 중간 ZIP 파일은 실질적인 작업 흐름을 끊는 요소가 될 수 있다.

### 4. ZIP 생성 후 복사폴더를 삭제하는 기존 구조 자체는 장점이 있음

현재처럼 ZIP 파일을 만든 뒤 중간 복사폴더를 삭제하는 구조는 런타임 공간에 불필요한 대용량 복사본이 남지 않는다는 장점이 있다.

즉, 단순히 `check.html` 결과만 사용자에게 제공하는 목적이라면 기존 ZIP 방식은 정리 측면에서 합리적이다.

## 정리

ZIP 방식은 단독 결과 제공에는 깔끔하지만, `check.html -> qa.html -> WBS 후처리`로 이어지는 통합 흐름에서는 중간에 압축/해제 과정이 추가되어 비효율적이다.

반대로 ZIP 생성을 제거하고 복사폴더를 유지하면 후속 작업 연계는 쉬워지지만, `덤프 후 반영`을 반복 실행했을 때 결과 폴더가 누적되는 관리 이슈가 생긴다.

따라서 향후 개선 시에는 복사폴더를 단순 임시폴더가 아니라 후속 작업용 공식 결과폴더로 취급하고, 필요 시 오래된 결과폴더를 정리하거나 덮어쓰기/백업 정책을 추가하는 방향을 검토한다.

## 반영 결정

반복 실행 시 기존 결과폴더를 덮어쓰지 않고, 검사 폴더명 뒤에 버전을 붙여 새 결과폴더를 만든다.

예시:

```text
테스트_v0.1
테스트_v0.2
테스트_v0.3
```

이 방식은 사용자가 여러 번 `덤프 후 반영`을 실행하더라도 각 실행 결과를 구분할 수 있고, 동시에 원본 폴더와 이전 결과폴더를 직접 훼손하지 않는다.

## 추가 반영

`check.html`의 `덤프 후 반영` 결과 폴더는 `qa.html`로 전달할 수 있도록 처리한다.

기본 흐름:

```text
check.html
-> 덤프 후 반영
-> 결과 폴더 유지
-> QA 생성으로 이동
-> qa.html에서 결과 폴더 기준으로 QA 산출물 생성
-> 기존 QA 산출물은 같은 폴더의 bak 하위로 이동
-> 생성 QA 산출물을 기존 위치/파일명으로 배치
```

현재 자동 탐색 대상은 다음과 같다.

```text
사용자인터페이스설계서 PDF
단위시험케이스 HWPX
통합시험시나리오 XLSX
```

## QA 매칭 기준 보정

사용자인터페이스설계서는 설계 산출물이므로 `check.html`에서 만든 시험/인도 결과 폴더 안에 없을 수 있다. 따라서 `qa.html`에서는 화면/사용자인터페이스설계서를 직접 업로드받는다.

지원 형식:

```text
HWP
HWPX
PDF
```

QA 생성은 단일 파일 묶음이 아니라 요구사항 ID 기준으로 수행한다.

예시:

```text
SFR-ESS-001 사용자인터페이스설계서
SFR-ESS-001 단위시험케이스
SFR-ESS-001 통합시험시나리오
```

위 세 파일이 모두 존재할 때 `SFR-ESS-001`에 대한 QA 생성/교체 작업을 수행한다. 다른 요구사항 ID도 같은 방식으로 독립적으로 매칭한다.
