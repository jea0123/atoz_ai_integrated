# 현재 기능 정리

이 문서는 현재 코드 기준으로 앱이 제공하는 주요 기능을 읽기 쉽게 정리한 것이다.

## 1. 산출물 매핑

위치:
- 화면: `web/check.html`
- API: `POST /api/check`
- 주요 코드: `output_file_check/folder_workflow.py`, `output_file_check/folder_mapping.py`, `output_file_check/folder_matching.py`

기능:
- 문서관리표준 파일과 대상 폴더를 받아 산출물 목록을 읽는다.
- 문서관리표준의 산출물명, 파일명, 폴더 경로 표를 기준으로 대상 폴더 안 파일을 매칭한다.
- 산출물 ID는 반영 후 새 문서번호/파일명에 쓸 값이며, 기존 파일 매칭 기준으로 쓰지 않는다.
- 매칭 모드는 기본 규칙 기반이며, 설정에 따라 AI 보강 모드를 사용할 수 있다.
- `bak`, `backup`, `백업`, `임시`, `temp`, `tmp` 폴더는 기본적으로 매칭 대상에서 제외한다.
- `원본` 폴더는 경로 비교에서 투명 폴더처럼 취급할 수 있다.
- 파일명, 경로, 표지/본문에서 읽은 문서명을 함께 사용해 후보 점수를 계산한다.
- 표지/본문 문서명 비교에서는 괄호 안 내용을 제거한 버전도 함께 비교한다.
  - 예: `성능(부하)시험결과서`는 표지 제목 비교에서 `성능시험결과서`와도 비교된다.
  - 파일명/경로 비교에는 이 괄호 제거 전처리를 적용하지 않는다.

출력:
- 매칭된 산출물 목록
- 미매칭 산출물 목록
- 후보 파일과 점수/사유
- 요청 ID와 스캔 파일 수

## 2. 산출물 일괄 반영

위치:
- 화면: `web/check.html`
- API: `POST /api/folder-apply`
- CLI: `folder_apply.py`
- 주요 코드: `output_file_check/folder_apply_ops.py`

기능:
- 원본 폴더를 직접 수정하지 않고 `web_runtime/results/folder-dumps` 아래에 버전 폴더로 복사한다.
  - 예: `원본폴더_v0.1`, `원본폴더_v0.2`
- 매칭된 파일에 문서관리표준 기준의 산출물 ID, 산출물명, 사업명/프로젝트명을 반영한다.
- 파일명도 표준 산출물 ID와 산출물명 기준으로 변경한다.
- 기본 파일명 버전은 `v0.1`로 맞춘다.
- 표지와 머릿말의 버전 값도 `v0.1` 기준으로 반영한다.
- HWP/HWPX, Excel, PowerPoint 계열 문서를 대상으로 처리한다.
- `Thumbs.db`, `desktop.ini`, Office 임시 파일 같은 노이즈 파일은 업로드/복사/결과 정리 단계에서 제외 또는 제거한다.
- 반영 결과 README를 결과 폴더 안에 만든다.

현재 정책:
- `bak`, `backup`, `백업` 폴더에 반영 전 원본을 복사하는 기능은 제거했다.
- `bak` 안 파일을 보조 후보로 다시 읽는 기능도 제거했다.
- 남아 있는 `bak` 관련 설정은 “건드리지 않기 위한 ignore 목록”이다.

## 3. 요구사항별 자동 생성

위치:
- 화면: `web/check.html`의 요구사항 파일 입력
- 주요 코드: `output_file_check/requirement_generation.py`

기능:
- 요구사항 파일명에서 `SFR-...` 형식의 요구사항 ID를 추출한다.
- 관리산출물 화면에서는 제안요청서 본문을 읽고 요구사항목록표 주변의 `SFR-...` 고유번호를 요구사항 ID로 추출할 수 있다.
- ID 형식은 고정값이 아니다.
  - 예: `SFR-001`, `SFR-ESS-001`, `SFR-IIL-003`, `SFR-SFD-002`
- 산출물 반영이 끝난 파일을 템플릿으로 삼아 요구사항 ID별 파일을 복제 생성한다.
- 기본 대상 산출물에는 인터뷰결과서, 업무정의서, 사용자인터페이스설계서, 데이터 설계 문서, 시험 케이스/시나리오/결과서, 프로그램목록, 인수인계시험결과서 등이 포함된다.
- 관리산출물 화면에서는 이번에 반영된 파일 중 파일명 뒤쪽에 `SFR-...` 요구사항 ID가 이미 붙어 있는 파일만 복제 대상으로 본다.
- 제안요청서에서 추출한 ID는 기존 파일명 tail의 `SFR-...` 값을 치환할 때만 사용하며, ID가 없는 대장류 파일에는 새 ID별 파일을 만들지 않는다.
- `[별첨1]인수인계확인서` 같은 별첨 파일은 인수인계시험결과서의 특수 케이스로 보존한다.
- 파일명 안의 요구사항 ID는 새 요구사항 ID로 치환한다.
- 파일명 안의 버전은 `v0.1`로 정규화한다.
- 같은 산출물의 기본 파일과 별첨 파일은 서로 다른 템플릿 변형으로 보고 각각 생성한다.
- 중복 템플릿은 변형 기준으로 묶어서 불필요한 교차 생성과 타임스탬프 파일 생성을 줄인다.

정리 정책:
- 생성에 사용한 템플릿 원본 파일은 삭제한다.
- 같은 산출물 폴더 안에서 요구사항 ID가 붙은 이전 생성 파일 중 이번 결과물이 아닌 파일은 삭제 대상으로 본다.
- 이번에 생성 또는 대체된 결과 파일은 유지한다.
- 다른 산출물명/산출물 ID로 보이는 파일은 삭제하지 않는다.

프로그램소스 폴더:
- 요구사항 ID별 빈 폴더를 `04.구현 > 01.어플리케이션개발 > 01.프로그램개발 > 02.프로그램소스` 아래에 만든다.
- 이 프로그램소스 경로 안의 기존 파일/폴더는 지우고 새 요구사항 ID 폴더만 만든다.
- 다른 경로의 빈 폴더를 자동 생성하거나 삭제하지 않는다.

출력:
- `README_요구사항별_자동생성.md`
- 생성 파일 목록
- 삭제한 기존 기준/이전 생성 파일 목록
- 요구사항 ID가 없는 입력 파일 목록
- 오류/경고 목록

## 4. 메타데이터 반영

위치:
- 화면: `web/metadata.html`
- API: `POST /api/metadata-preview`, `POST /api/metadata-apply`
- 주요 코드: `document_update/metadata_workflow.py`, `document_update/metadata_update.py`

기능:
- WBS 파일과 문서관리표준을 입력받는다.
- WBS에서 산출물별 담당자, 개정일자 등 메타데이터를 읽는다.
- 문서관리표준 표지에서 승인/작성자 기준 값을 읽는다.
- 결과 폴더 또는 직접 업로드한 폴더의 문서를 WBS 산출물과 매칭한다.
- 미리보기에서 matched, ambiguous, unmatched 상태를 확인할 수 있다.
- 적용 단계에서 표지와 개정이력 영역의 작성자/일자/담당자 정보를 갱신한다.
- 기존 산출물 매핑 결과 폴더 경로를 받아 후처리할 수 있다.
- 반영 결과 폴더는 zip 다운로드를 제공한다.

## 5. QA 산출물 생성

위치:
- 화면: `web/qa.html`
- API:
  - `POST /api/generate-tc`
  - `POST /api/generate-ts`
  - `POST /api/run-qa-folder`
- 주요 코드: `qa_generation/generate_tc.py`, `qa_generation/generate_ts.py`, `qa_generation/folder_pipeline.py`

기능:
- 사용자인터페이스설계서에서 화면 ID, 요구사항 ID, 처리 흐름을 읽어 단위시험케이스를 생성한다.
- 단위시험케이스와 UI 설계서를 이용해 통합시험시나리오를 생성한다.
- 폴더 QA 파이프라인에서는 산출물 매핑 결과 폴더를 기준으로 요구사항 ID별 입력 세트를 찾는다.
- 요구사항 ID별로 다음 파일 묶음을 찾아 처리한다.
  - 사용자인터페이스설계서
  - 단위시험케이스 템플릿
  - 단위시험결과서 템플릿
  - 통합시험시나리오 템플릿
  - 통합시험결과서 템플릿
- 생성된 QA 파일을 기존 결과 폴더의 알맞은 위치에 배치한다.
- QA 처리 중 요청 취소를 지원한다.
- 결과 파일은 다운로드 토큰을 통해 내려받을 수 있다.

## 6. 공통 런타임/웹 기능

위치:
- 서버: `web_app.py`
- 공통 런타임: `app_runtime.py`
- 업로드 처리: `web_uploads.py`
- 취소 처리: `cancellation.py`

기능:
- `check`, `metadata`, `qa` 화면을 제공한다.
- multipart 업로드 파일을 임시 작업 폴더에 저장한다.
- 업로드 경로를 안전한 상대 경로로 정리한다.
- 결과 폴더 zip 다운로드를 제공한다.
- 일회성 다운로드 파일은 다운로드 후 정리할 수 있다.
- 장시간 처리 요청은 request ID 기준으로 취소할 수 있다.
- `.env`와 요청 필드에서 Ollama URL, 모델명, 런타임 모드를 결정한다.
- 임시 작업 폴더와 결과 폴더를 관리한다.

## 7. CLI 기능

위치:
- `folder_apply.py`

기능:
- 웹 화면 없이 산출물 매핑만 실행할 수 있다.
- 웹 화면 없이 산출물 일괄 반영까지 실행할 수 있다.
- 문서관리표준 파일을 직접 지정하거나 대상 폴더 안에서 자동 탐색한다.
- ignore 폴더, transparent 폴더, 매칭 모드, AI URL/모델 등을 인자로 받을 수 있다.

## 8. 현재 주의할 점

- 서버가 이미 떠 있으면 코드 수정 후 서버 재시작이 필요하다.
- 브라우저 폴더 업로드는 빈 폴더를 보내지 않는 경우가 있어, 일반 빈 폴더 자동 복원은 하지 않는 정책으로 정리했다.
- `bak`, `backup`, `백업` 폴더는 기본적으로 처리 대상이 아니다.
- 요구사항별 자동 생성은 “업로드한 요구사항 파일명에서 추출된 ID”를 기준으로 동작한다.
- 요구사항 ID는 고정 목록이 아니며 실행 때마다 달라질 수 있다.
- 결과 폴더에 이전 실행 결과가 섞여 있으면, 요구사항별 자동 생성 대상 산출물 폴더에서는 같은 산출물의 이전 요구사항 생성 파일을 정리한다.
